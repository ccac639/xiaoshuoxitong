# 个人版 AI Novel OS v2.0 Lite

> 副标题：**一个作者 · 一台电脑 · 一部百万字小说**
> 本文是 v2.0（docs/架构设计-v2.0-后端重构.md）的**瘦身版**。
> 保留五大核心思想，砍掉一切企业级/分布式过度设计。
> **不含实现代码**，仅设计。

---

## 0. 定位与边界

### 0.1 一句话

> 这不是 SaaS，是**跑在你自己电脑上的 AI 写作操作系统**。
> `docker compose up` → 打开浏览器 → 开始写百万字。

### 0.2 目标（做什么）

| 目标 | 说明 |
|---|---|
| ✅ 长篇不崩 | 100 万字后剧情仍自洽 |
| ✅ 人物不崩 | 人物状态由事件推导，永不遗忘 |
| ✅ 世界观一致 | 规则/设定变更可追踪、冲突可检测 |
| ✅ AI 味降低 | 专门的 `ai_remove` / 风格 Skill |
| ✅ 节省 Token | Token Manager 全程预算控制 |
| ✅ 持续写 100 万字 | 分层记忆 + 事件溯源撑住长程一致性 |

### 0.3 非目标（明确不做）

❌ 多用户 / 多租户　❌ 高并发　❌ 商业化插件市场
❌ 企业权限系统　❌ Kubernetes / 分布式集群　❌ 分布式事件总线

## 1. 简化后的整体架构

```
        Next.js 前端（保留现有 UI）
                 │  HTTP
        FastAPI 后端（单进程）
                 │
   ┌─────────────┼─────────────┐
   │             │             │
PostgreSQL     Redis        本地文件
(+pgvector)  (当前上下文)   (Markdown 正文)
                 │
        ┌────────┴────────┐
        │   AI Engine     │
        │  Agent · Skill  │
        │ Memory · World  │
        │  Token Manager  │
        └─────────────────┘
```

**关键取舍**：后端**单进程**即可，不拆微服务；异步用**简单后台线程**，不上 Celery。

## 2. 数据库（PostgreSQL 一个库，够用）

> 不拆库、不拆表拆得太碎。6 张核心表 + 3 张向量表。

### 2.1 核心表

```sql
-- 小说项目
novel (
  id        UUID PK,
  title     TEXT,
  genre     TEXT,
  status    TEXT,          -- draft / writing / done
  created_at TIMESTAMPTZ
)

-- 章节（正文存文件，库里只存路径+摘要）
chapter (
  id           BIGSERIAL PK,
  novel_id     UUID,
  chapter_no   INT,
  title        TEXT,
  content_path TEXT,       -- data/novels/<id>/chapters/00123.md
  summary      TEXT,
  created_at   TIMESTAMPTZ
)

-- 故事事件（★核心：事件溯源的真相）
story_event (
  id          BIGSERIAL PK,
  novel_id    UUID,
  chapter_id  BIGINT,
  seq         BIGINT,      -- 小说内单调递增
  event_type  TEXT,        -- 人物变化/关系/世界规则/伏笔埋收...
  impact_json JSONB,       -- 结构化增量
  description TEXT,
  created_at  TIMESTAMPTZ
)

-- 世界状态（★核心：事件的投影结果）
world_state (
  novel_id  UUID PK,
  state_json JSONB,        -- 人物/组织/地点/规则/时间线/伏笔
  version   INT
)

-- 人物状态（合并版，个人版不拆一堆表）
character_state (
  novel_id     UUID,
  character_id TEXT,
  data_json    JSONB,      -- 属性/关系/秘密/物品/技能/弧线
  PRIMARY KEY (novel_id, character_id)
)

-- Skill 执行记录（追踪：用了啥Skill、花了多少Token、结果如何）
skill_execution (
  id          BIGSERIAL PK,
  novel_id    UUID,
  chapter_id  BIGINT,
  skill_name  TEXT,
  token_cost  INT,
  result_json JSONB,
  created_at  TIMESTAMPTZ
)
```

### 2.2 向量表（pgvector，不上 Milvus）

```sql
chapter_embedding   (chapter_id, novel_id, embedding vector)   -- 历史章节语义召回
summary_embedding   (chapter_id, novel_id, embedding vector)   -- 摘要召回
character_embedding (character_id, novel_id, embedding vector) -- 人物一致性检索
```

> 个人百万字，pgvector 完全够。写到 **500 万字+** 再考虑迁 Milvus。

## 3. Event Sourcing（保留思想，做轻）

> 保留「事件是真相、状态是投影」的思想，但**不做 Event Bus / Kafka / 消息队列**。

### 3.1 简化流程（同步，一条链）

```
生成章节
   ↓
提取事件（从章节内容 → story_event）
   ↓
保存 event（追加，不改历史）
   ↓
更新 world_state（把 impact_json 应用上去，version+1）
   ↓
保存
```

- 就是一次函数调用里顺序做完，**无异步总线、无 Projection 服务、无 Snapshot 集群**。
- 「回滚/改剧情」个人版做法：截断某 `seq` 之后的事件 → 从头把 `impact_json` 重新叠加算出 `world_state`（百万字量级单机秒级可算，不需要 Snapshot 机制；真慢了再加缓存快照）。

### 3.2 保留 vs 砍掉

| v2.0 完整版 | Lite 版 |
|---|---|
| Event Bus | ❌ → 直接函数调用 |
| Projection 服务 | ❌ → 更新时直接写 world_state |
| Snapshot 表 | ⚠️ 暂不做，需要时再加 |
| Replay | ✅ 保留（重算 world_state），但按需触发 |
| story_event 表 | ✅ 核心保留 |

## 4. Memory（三级 → 两级半）

| 级别 | 存储 | 内容 |
|---|---|---|
| **一级：当前上下文** | Redis | 当前章节、当前出场人物、当前冲突、Token 余量 |
| **二级：剧情数据库** | PostgreSQL | 章节摘要、事件链、人物变化、伏笔 |
| **三级：语义检索** | pgvector（不是 Milvus） | chapter / summary / character embedding |

**Memory Agent 检索路由**（个人版够用）：
- 要人物/关系 → 直接读 `character_state`（精确不丢）
- 要近期剧情 → 读近 K 章 `summary` + Redis 当前上下文
- 要相似片段/伏笔原文 → pgvector 相似度检索
- 组装时受 Token Manager 预算约束

---

## 5. Agent（6 个 → 4 个）

```
1. Planner  下一章规划 · 冲突 · 爽点 · 伏笔
2. Writer   正文生成
3. Memory   找历史资料 · 组装上下文
4. Audit    人物检查 · 逻辑检查 · AI味检查
```

- **砍掉 World Agent / Editor Agent**：
  - World 更新逻辑并入「章节保存后处理」（提取事件→更新 world_state），不需要独立 Agent。
  - Editor 的修订并入 Writer 的二次生成或 Audit 反馈循环。
- **砍掉 Agent Registry**：4 个 Agent 直接是 4 个 Python 模块/函数，不做动态注册。

### 5.1 主流程闭环

```
用户点「开始创作」
   ↓
Planner   → 下一章大纲（冲突/爽点/待埋伏笔）
   ↓
Memory    → 组装上下文（人物状态 + 近期摘要 + 相似片段）
   ↓
Writer    → 生成正文
   ↓
Audit     → 人物/逻辑/时间线/AI味检查（有问题回 Writer 修）
   ↓
保存章节 → 提取事件 → 更新 world_state
```

> LangGraph **后期再上**（把这 4 步编译成 StateGraph）；先用普通函数编排跑通。

## 6. Skill 系统（保留，去平台化）

> 不做 Skill 市场 / 版本管理 / 动态加载。**目录 + 一个 yaml** 即可。

### 6.1 目录

```
skills/
├── writer/          # 写作
├── polish/          # 润色
├── ai_remove/       # 去 AI 味
├── character_check/ # 人物一致性
├── fanqie_style/    # 番茄风格
└── qidian_style/    # 起点风格
```

### 6.2 一个 skill.yaml（够用）

```yaml
name: character_check
prompt: prompt.md          # 或直接内联 prompt 文本
input:  [chapter, world_state]
output: [issue_list]
```

- 启动时扫 `skills/` 读 yaml，按名调用。**无 version、无 depends_on、无 triggers 编排引擎**。
- 你已有的 `fanqie-novel-skill`、Truth Files、27 维审计 → 直接落成 `fanqie_style/` 和 `character_check/` 的 prompt 资产。

## 7. Token Manager（必须保留，做轻）

> 核心需求，但不做企业级预算引擎。一个 `token_manager.py` 搞定。

### 7.1 职责

- 记录三个维度：**小说总消耗 / 单章消耗 / 各模型消耗**（落 `skill_execution.token_cost` + 一张汇总）。
- 预算提示：
  ```
  一本小说预算：200 万 tokens
  当前已用：  53 万
  剩余：      147 万
  ```
- 调用前预估，接近上限时**先压缩上下文再调用**（减少 pgvector 召回数 / 摘要粒度降级）。
- 与 ModelRouter 协同：Planner 用强模型、Audit 用便宜模型。

### 7.2 砍掉

❌ 滑动窗口限流集群　❌ 多层配额分布式协调　❌ 独立 Governor 服务
→ 就是后端里一个模块，函数调用时 `token_manager.guard()` 包一下。

## 10. 从 v2.0 完整版砍掉的清单

| 砍掉 | 原因 | 替代 |
|---|---|---|
| ❌ Milvus | 个人百万字用不上 | pgvector |
| ❌ Celery | 个人电脑没必要 | 后台线程 |
| ❌ Event Bus | 过重 | 直接函数调用 / Redis 简单队列 |
| ❌ Agent Registry | 过度设计 | 4 个模块直接调 |
| ❌ MinIO | 单机没必要 | 本地文件系统 |
| ❌ Snapshot 集群 | 量级不需要 | 需要时按 seq 重算 |
| ❌ 多租户/权限 | 单用户 | 完全去掉 |
| ❌ World/Editor Agent | 可合并 | 并入保存后处理 / Writer 循环 |

## 11. 必须保留的五大核心（★★★★★）

> 这 5 个才是「百万字不崩」的真正核心，一个都不能少：

1. **Event Sourcing 思想** — story_event 是真相，world_state 是投影
2. **World State** — 人物/世界由事件推导，可查可检冲突
3. **Memory Layer** — 三级检索，长程不遗忘
4. **Skill Runtime** — 写作/润色/去AI味/风格 插件化（目录版）
5. **Token Governor** — 全程成本控制，百万字可负担

## 12. 开发阶段规划（Lite）

| Phase | 目标 | 验证 |
|---|---|---|
| **0** | Docker Compose（PG+pgvector+Redis）+ FastAPI 脚手架 | `docker compose up` → `/health` 200 |
| **1** | 核心表 + story_event 读写 + world_state 更新（同步流程） | 追加事件 → world_state 正确变化 |
| **2** | Memory 两级半（Redis + PG 摘要 + pgvector） | 第 100 章能召回第 1 章片段 |
| **3** | 4 Agent 函数编排跑通闭环 | 点「开始创作」出带 reason 的章节 |
| **4** | Skill 目录加载（迁 fanqie/character_check/ai_remove） | 加一个 skill 目录即生效 |
| **5** | Token Manager（预算/统计/压缩降级） | 超预算自动压缩，成本可查 |
| **6** | 接真实 LLM（含本地模型）+ LangGraph 化 | 脱离 Mock，产出真实文本 |

## 13. 一句话收尾

v2.0 完整版是「如果要做成平台该怎么设计」；
**Lite 版是「你一个人写百万字，实际需要的最小强内核」。**
砍掉的是规模，保留的是**让小说 100 万字不崩的那 5 根骨头**。
最终形态：`docker compose up` → 你的个人 AI 写作工作站。
