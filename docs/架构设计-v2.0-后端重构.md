# xiaoshuoxitong 后端重构架构设计 v2.0

> 副标题：**AI Novel OS — Story Engine 而非 AI Writer**
> 本文档是 v1.0 架构设计（docs/架构设计-百万字AI创作平台.md）的**升级版**，聚焦六个核心支柱：
> Event Sourcing 数据模型 · World State Engine · Memory Layer · Agent Runtime · Skill Runtime · Token Governor。
> **不含实现代码**，仅设计。

---

## 0. 范式转换（本文档的底层假设）

### 0.1 一句话定位

> 百万字 AI 小说创作，本质是一个**游戏服务器**。
> 玩家动作 → 世界变化 → 保存状态 → 继续运行；
> 小说剧情事件 → 人物变化 → 世界变化 → 生成章节。
> **完全同构。**

### 0.2 旧范式 vs 新范式

| 维度 | v1.0（文本范式） | v2.0（世界状态范式） |
|---|---|---|
| 小说是什么 | 一串章节文本 | 一个持续演化的**世界状态** |
| 数据库职责 | CRUD 存 chapter/content/summary | **事件溯源**：事件日志是唯一真相，状态是投影 |
| 一致性来源 | 作者/AI 自己记着 | **由事件流重算**，可回滚、可重放 |
| AI 角色 | 一个「写手」 | Planner / Writer / Memory / World / Audit 多 Agent 协作 |
| 核心难点的应对 | 「尽量别忘」 | 「忘不了」——状态永远由事件推导 |

### 0.3 核心原则（贯穿全文）

1. **事件溯源（Event Sourcing）**：所有剧情变动先落事件，状态是事件的有序投影。
2. **CQRS**：写侧只追加事件；读侧是物化视图（快照 / 摘要 / 向量）。
3. **状态可重算**：任意历史时刻的世界状态 = 从 0 重放事件到该点。
4. **Token 是硬约束**：一切上下文组装在 Token Governor 预算内完成，超预算即压缩降级。
5. **Skill 即插件**：写作/润色/审计/一致性都是可独立版本化、可组合的 Skill。

## 1. Event Sourcing 数据模型

> 这是百万字系统**最关键的支柱**。没有它，100 万字后人物必崩、伏笔必丢。

### 1.1 核心思想

- **事件日志（Event Log）= 唯一真相（Source of Truth）**。
- 世界状态 **不是**被「覆盖写入」的，而是**对事件流的有序投影（Projection）**。
- 例：人物「萧炎」当前境界不是一行 `level=斗皇` 的字段，而是：
  - 事件① 加入宗门 → 事件② 背叛宗门 → 事件③ 重修功法 → …… **重放后得到**当前境界。

### 1.2 事件表（PostgreSQL：`story_event`）

```sql
story_event (
  id            BIGSERIAL PK,
  novel_id      UUID        NOT NULL,        -- 所属小说
  chapter_id    BIGINT      NULL,            -- 触发章节（初始化/补录事件可空）
  seq           BIGINT      NOT NULL,        -- 小说内全局单调递增序号（乐观锁/排序）
  event_type    VARCHAR     NOT NULL,        -- 见 1.3 枚举
  description   TEXT        NOT NULL,        -- 自然语言描述（便于审计/回放阅读）
  importance    SMALLINT    NOT NULL DEFAULT 2,  -- 1~5，影响检索权重与快照频率
  impact_json   JSONB       NOT NULL,        -- 结构化增量，见 1.4
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  actor_agent   VARCHAR     NULL,            -- 触发事件的 Agent（human/planner/world...）
  UNIQUE (novel_id, seq)
);
```

### 1.3 事件类型枚举（`event_type`）

| 类型 | 含义 | 典型 impact |
|---|---|---|
| `CHARACTER_STATE_CHANGE` | 人物属性/心理/秘密变化 | character_deltas |
| `RELATIONSHIP_CHANGE` | 人物间关系增减 | relationship_deltas |
| `WORLD_RULE_CHANGE` | 世界规则/力量体系变动 | world_deltas.rules |
| `LOCATION_DISCOVERY` | 新地点/地图解锁 | world_deltas.locations |
| `FACTION_CHANGE` | 势力兴衰/易主 | world_deltas.factions |
| `ITEM_OBTAINED` | 获得/失去物品/异火/功法 | character_deltas.inventory |
| `SKILL_LEARNED` | 习得技能/天赋 | character_deltas.skills |
| `FORESHADOW_PLANT` | 埋设伏笔 | plot_flags.{key: "planted"} |
| `FORESHADOW_PAYOFF` | 伏笔回收 | plot_flags.{key: "paid"} |
| `PLOT_FLAG_SET` | 剧情开关/约束生效 | plot_flags.{key: value} |
| `TIMELINE_ADVANCE` | 时间线推进（年/月/季） | world_deltas.timeline |
| `CHARACTER_INTRO` / `CHARACTER_EXIT` | 人物登场/退场 | character_deltas.present |

### 1.4 `impact_json` 结构（状态增量契约）

所有 delta 都是**增量（patch）**，投影引擎负责应用。

```json
{
  "character_deltas": {
    "xiao_yan": {
      "level": "+1",                 // 相对增量或绝对值由字段语义决定
      "psychology": "revenge",       // 覆盖式字段
      "secrets": ["knows_earth_fire_location"],
      "inventory_add": ["异火·青莲地心火"],
      "skills_add": ["焚诀·进阶"]
    }
  },
  "relationship_deltas": {
    "xiao_yan__yao_lao": "+80",      // 关系键用 __ 连接，数值为增减量
    "xiao_yan__yun_yun": "-10"
  },
  "world_deltas": {
    "stage": "entered_new_phase",
    "rules_add": ["斗气之上尚有斗尊"],
    "factions": { "星陨阁": "rising" }
  },
  "plot_flags": {
    "earth_fire_arc": "active",
    "master_revenge": "planted"
  }
}
```

### 1.5 投影（Projection）与快照（Snapshot）

- **投影引擎**：顺序消费 `story_event`，把 `impact_json` 应用到内存状态对象，产出 `world_snapshot`。
- **为什么需要快照**：百万字 = 数十万事件，每次全量重放太慢。
  - 每 N 章（可配，默认 10 章）或每 M 个高 importance 事件，落一个 `world_snapshot`（见 §2.4）。
  - 重放时从最近快照 + 其后增量事件开始，**O(增量)** 而非 O(全量)。
- **事件不可变**：事件一旦写入**禁止 UPDATE/DELETE**（防篡改真相）。修正剧情 = 追加一条反向/纠正事件（如 `RELATIONSHIP_CHANGE -80`）。

### 1.6 事件溯源带来的能力（直接回应你的诉求）

| 你的诉求 | 事件溯源如何满足 |
|---|---|
| 第 100 章还记得第 1 章 | 状态由全量事件投影，永不「忘记」 |
| 人物不崩 | 人物状态=事件推导；矛盾事件触发 Audit 告警 |
| 世界规则不乱 | 规则变更是事件；规则冲突被 World Agent 检测 |
| 伏笔回收 | `FORESHADOW_PLANT` ↔ `FORESHADOW_PAYOFF` 配对追踪，未回收可视化 |
| 可回滚/重算/改剧情 | 截断事件流或插入新事件后重放即可 |

## 2. World State Engine（世界状态引擎）

> 负责把**事件流**变成**可读、可查、可推演的世界状态**，并检测一致性冲突。

### 2.1 职责边界

- 订阅 Event Bus 的 `story_event`（来自 Writer/Audit/Human）。
- 维护并物化：`characters` / `organizations` / `locations` / `rules` / `timeline` / `plot_flags`。
- 对外提供**读模型**（World State Panel 直接消费）。
- **冲突检测**：当新事件与既有状态矛盾（如「已死人物」又触发 `CHARACTER_STATE_CHANGE`），交 Audit Agent。

### 2.2 人物状态（不是人物卡，是状态机）

```json
character_state {
  "id": "xiao_yan",
  "name": "萧炎",
  "present": true,
  "attributes": {                         // 可被事件覆盖/增量
    "level": "斗皇",
    "psychology": "revenge",
    "health": "injured"
  },
  "relationships": {                      // 关系图
    "yao_lao": 80,
    "yun_yun": -10
  },
  "secrets": ["knows_earth_fire_location"],
  "inventory": ["异火·青莲地心火", "焚诀"],
  "skills": ["焚诀·进阶", "佛怒火莲"],
  "known_facts": ["药老真实身份=魂殿追杀对象"],
  "arc": {                               // 人物弧线（供 Memory 检索）
    "act1": "废柴逆袭",
    "act2": "寻异火"
  },
  "version": 42,                          // 对应快照版本
  "last_event_seq": 1337
}
```

### 2.3 世界实体（组织 / 地点 / 规则 / 时间线）

- `organizations`：`{ id, name, status(rising/falling/destroyed), members[], territory[] }`
- `locations`：`{ id, name, discovered_at_seq, connected_to[], secrets[] }`
- `rules`（力量体系 / 世界观约束）：`{ id, statement, source_event_seq, conflicts_with[] }`
- `timeline`：`{ current_epoch, advances: [{seq, delta}] }`
- `plot_flags`：`{ key: {status: planted|active|paid, planted_seq, paid_seq?} }`

### 2.4 快照表（`world_snapshot`，物化读模型）

```sql
world_snapshot (
  id           BIGSERIAL PK,
  novel_id     UUID     NOT NULL,
  version      INT      NOT NULL,         -- 递增
  base_seq     BIGINT   NOT NULL,         -- 该快照覆盖到的事件序号
  state_json   JSONB    NOT NULL,         -- 完整世界状态（压缩后）
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (novel_id, version)
);
```

### 2.5 与现有代码衔接

- 现有 `src/server/project/stateManagerV2.ts` 的 `getWorldState(id)` 是**单例内存 Map**，v2.0 改为：
  - 写：追加 `story_event` → 投影引擎更新内存态 → 周期落 `world_snapshot`。
  - 读：直接返回最近 `world_snapshot` + 内存增量，无需重放全量。
- 现有 `StoryMemoryV2` 的「结构化摘要」可作为 `plot_flags` / `character.arc` 的初始化数据源。

## 3. Memory Layer（分层记忆）

> 百万字稳定性 = 检索对了「该回忆什么」。记忆不是越大越好，是**分层 + 路由 + 预算内组装**。

### 3.1 三层架构

| 层 | 存储 | 内容 | 生命周期 | 检索方式 |
|---|---|---|---|---|
| **L0 短期** | Redis | 当前章节工作上下文、激活的 plot_flags、本批次事件 delta、Token 预算余量 | 章节级（生成完即清/归档） | 直接取 |
| **L1 中期** | PostgreSQL | 章节摘要、事件链（story_event 投影）、人物弧线、伏笔状态 | 永久（结构化） | SQL + 索引 |
| **L2 长期** | Milvus（开发期 pgvector） | 全文语义块、人物/世界观 embedding、历史章节向量 | 永久（向量） | ANN 相似度 |

### 3.2 检索路由（Memory Agent 核心逻辑）

```
Memory Agent 接收查询意图（如「写下一章需要回忆萧炎与药老的关系」）
  → 解析需要的记忆类型：
      - 关系/状态    → 直接读 L0/L1 的 character_state（精确，不丢）
      - 近期剧情     → L1 章节摘要（近 K 章）+ L0 当前上下文
      - 语义相似     → L2 向量检索（与当前情节相似的过往片段、伏笔原文）
  → 按 Token Governor 预算组装上下文窗口
  → 返回 { context_pack, citations[], token_cost }
```

### 3.3 上下文组装与压缩

- **重要性打分**：事件 `importance` + 人物出场频率 + 伏笔活跃度 → 决定哪些进上下文。
- **层级摘要**：L1 章节摘要本身是 L2 全文的压缩；L2 又可在 L1 之上再抽象。
- **增量更新**：新章节生成后，只更新受影响的摘要/向量块，不重建全部。
- **淘汰（Eviction）**：超预算时按 `importance × recency` 降序保留，其余丢弃或降级到更粗摘要。

### 3.4 与现有代码衔接

- 现有 `StoryMemoryV2`（`src/server/memory/storyMemoryV2.ts`）的 `searchMemory`/`addMemory` 直接映射为 L1 摘要层接口。
- 向量检索在开发期用 `pgvector`（给 `chapters` 加 `embedding vector` 列），生产切 Milvus（§7 迁移路径）。

## 4. Agent Runtime（Agent 运行时）

> 多 Agent 协作的**编排与执行底座**，而非单个写手。

### 4.1 核心流水线（你的 Story Engine 闭环）

```
用户意图
  │
  ▼
[Event Flow Engine]  ── 捕获/生成剧情事件
  │
  ▼
[Planner Agent]      ── 下一阶段剧情、冲突设计、节奏控制
  │
  ▼
[Skill Runtime]      ── 选择并执行写作/润色/审计 Skill
  │
  ▼
[Memory Engine]      ── 检索过去、组装上下文（§3）
  │
  ▼
[World State Engine] ── 计算人物/世界状态变化（§2）
  │
  ▼
[Writer Agent]       ── 生成章节正文
  │
  ▼
[Audit Agent]        ── 检查人物/逻辑/时间线/AI味
  │
  ▼
保存章节 + 追加 story_event
```

### 4.2 Agent 角色契约

| Agent | 输入 | 输出 | 触发 |
|---|---|---|---|
| **Planner** | world_state + 目标 + 节奏参数 | 下一章大纲 / 冲突点 / 待埋伏笔 | 用户点「开始创作」 |
| **Writer** | context_pack + 大纲 + Skill 产物 | 章节草稿 | Planner 完成 |
| **Memory** | 查询意图 | context_pack（带引用） | 各 Agent 按需调用 |
| **World** | 新事件 / 草稿 | 状态增量 / 冲突告警 | 事件产生时 |
| **Audit** | 草稿 + world_state | 问题清单（人物/逻辑/时间线/AI味） | Writer 完成 |
| **Editor**（可选） | 草稿 + Audit 报告 | 修订稿 | Audit 发现问题 |

### 4.3 运行时底座（Agent Runtime Core）

- **Agent Registry**：注册可用 Agent（含其 LLM 角色、Token 配额、依赖）。
- **Event Bus**：Agent 间通过事件解耦通信（§6 Event Bus）。Writer 产出草稿 → 发 `CHAPTER_DRAFTED` → Audit 订阅处理。
- **Context Assembly**：每个 Agent 调用前，由 Memory Agent 在 Token Governor 预算内组装专属上下文。
- **并发与隔离**：Agent 调用 LLM 受 ModelRouter（§v1）四角色路由；失败重试、超时、熔断。
- **可观测**：每次 Agent 运行落 `agent_run` 记录（输入/输出摘要/耗时/Token/成本）。

### 4.4 编排实现路径

- **Phase 4 前**：用 Python 函数编排（Pipeline 顺序调用）跑通闭环，降低风险。
- **Phase 4**：将流水线编译为 **LangGraph `StateGraph`**，6 节点对应上表；每节点 `send_event` 到 Event Bus，前端 Panel 实时消费。
- 现有 `ChapterWriterV2` 的 6 阶段（plan→beats→draft→audit→revise→commit）是**天然 StateGraph 骨架**，直接迁移为节点。

## 5. Skill Runtime（技能运行时）

> Skill = 独立、可版本化、可组合的插件。不是「写作/润色/检查」三个写死函数。

### 5.1 目录结构（插件化）

```
skills/
  character_consistency/
    skill.yaml        # 契约
    prompt.md         # 提示词模板
    executor.py       # 可选：自定义逻辑（非纯提示词）
  polish_ai_taste/
    skill.yaml
    prompt.md
  hook_detector/      # 爽点检测
    skill.yaml
    prompt.md
  platform_qidian/    # 起点风格
    skill.yaml
    prompt.md
  platform_fanqie/    # 番茄风格
    skill.yaml
```

### 5.2 `skill.yaml` 契约

```yaml
name: character_consistency
version: 1.2.0
description: 检查生成章节与 world_state 的人物/关系一致性
triggers: [post_write, on_demand]
input:
  - chapter          # 待检章节正文
  - world_state      # 当前世界状态（来自 World State Engine）
  - character_ids    # 本章出场人物
output:
  - issue_list:      # [{character, type, severity, detail, suggestion}]
token_budget: 3000   # 该 Skill 单次最大 Token 消耗（受 Token Governor 约束）
depends_on: [world_state_engine]
model_role: audit    # 走 ModelRouter 的 audit 角色（见 v1）
timeout_ms: 30000
```

### 5.3 Runtime 职责

- **加载与校验**：启动时扫描 `skills/`，解析 `skill.yaml`，校验契约完整性。
- **调度**：根据 `triggers` 自动挂载（如 `post_write` → Writer 完成后自动跑 `character_consistency` + `hook_detector`）。
- **组合（Chain）**：Skill 可串联，例如 `writer → polish_ai_taste → character_consistency → hook_detector`，形成「生成管道」。
- **市场/可扩展**：新增平台风格 = 新增一个 `platform_xxx/` 目录，**零代码改动**接入。
- **版本与回滚**：`version` 字段 + 执行记录，可指定某章节用旧版 Skill 重跑。

### 5.4 与现有代码衔接

- 现有 `skillLibraryV2.ts` 的 Skill 契约（`id/name/description/category/trigger/prompt`）**直接升级为 `skill.yaml`**（字段一一对应），`prompt` 抽出为 `prompt.md`。这是**迁移而非重写**——它已是高质量资产。

## 6. Token Governor（Token 治理器）

> 百万字稳定创作的「节流阀」。一切上下文组装、LLM 调用都过它。

### 6.1 职责

1. **预算层级**：
   - 全局：单 novel 累计预算（防失控）。
   - 请求级：单次生成预算（如 1 章 ≤ 32k tokens）。
   - Agent 级：Planner/Writer/Audit 各占配额。
   - 窗口级：滑动窗口限流（防 API 抖动）。
2. **预估（Estimate）**：LLM 调用前先估算 prompt+expected completion tokens；超预算则**先压缩再调用**。
3. **压缩降级策略**（超预算时）：
   - L2 向量召回数 ↓ → L1 摘要粒度 ↓ → L0 短期上下文摘要化。
   - 仍超 → 拒绝并提示用户放宽预算或拆分任务。
4. **成本核算**：记录每次调用的 `model / role / agent / tokens / cost`，落 `token_usage`。
5. **与 ModelRouter 协同**：按角色选模型（planner 用强模型、audit 用便宜模型），在质量与成本间权衡。

### 6.2 状态（Redis）

```json
token_budget {
  "novel_id": "xxx",
  "total_limit": 2000000,
  "used": 1340000,
  "per_agent": { "planner": 80000, "writer": 600000, "audit": 120000 },
  "current_window": { "window_start": "...", "requests": 12, "tokens": 45000 }
}
```

### 6.3 与现有代码衔接

- 现有 `ModelRouter`（四角色成本模型）是 Token Governor 的**执行臂**；Governor 是**决策臂**。二者合体即为完整 Token 控制。
- 现有生成流程里 `callModel()` 是 Mock —— Phase 7 接入真实 LLM 时，所有调用统一经 `Governor.guard()` 包装。

## 7. 数据库 ER 再设计（状态系统视角）

> 从「CRUD 文本库」转为「事件 + 状态 + 向量 + 文件」四库协作。

### 7.1 PostgreSQL（主库，状态系统）

```
novels(id, title, genre, target_length, current_version, created_at)
chapters(id, novel_id, seq, title, summary, markdown_ref, status, created_at)
story_event(id, novel_id, chapter_id, seq, event_type, description, importance, impact_json, actor_agent, created_at)  -- 真相
world_snapshot(id, novel_id, version, base_seq, state_json, created_at)  -- 物化读模型
character_state(novel_id, character_id, state_json, version, last_event_seq)  -- 也可由快照派生，此处为热缓存
plot_flag(novel_id, key, status, planted_seq, paid_seq)  -- 伏笔追踪
skill_execution(id, novel_id, chapter_id, skill_name, version, input_ref, output_ref, token_cost, status, created_at)
agent_run(id, novel_id, agent, input_ref, output_ref, tokens, cost, duration_ms, created_at)
token_usage(id, novel_id, model, role, agent, prompt_tokens, completion_tokens, cost, created_at)
```

> 正文存 MinIO（`markdown_ref` 指向对象键），PG 只存元信息与引用——避免大文本拖慢关系库。

### 7.2 Milvus（向量，开发期 pgvector 替代）

4 个 Collection，均按 `novel_id` 分区（适配千万字）：
- `chapter_chunks`：历史章节语义块 → 相似剧情召回。
- `character_emb`：人物设定/弧线向量 → 人物一致性检索。
- `world_emb`：世界观/规则向量 → 设定检索。
- `summary_emb`：章节摘要向量 → 长程摘要召回。

### 7.3 Redis（缓存/队列/总线）

- 上下文缓存（L0）、Token 预算、异步任务队列、Event Stream（SSE 推送前端）、分布式锁。

### 7.4 MinIO（文件）

- 章节 Markdown、世界状态快照归档、OCR 导入文件、事件日志冷备份。

## 9. 开发阶段规划 v2（可运行、可验证）

| Phase | 目标 | 关键交付 | 验证标准 |
|---|---|---|---|
| **0** | 环境基建 | Docker Compose（PG+Redis+MinIO+pgvector）、FastAPI 脚手架、Alembic | `docker compose up` 后 `/health` 200 |
| **1** | Event Store | `story_event` 表 + 追加/查询 API + 投影引擎（内存） | 追加 100 事件，投影出正确 world_state |
| **2** | World State Engine | 人物/组织/地点/规则/伏笔状态 + 快照 + 冲突检测 | World State Panel 实时显示；埋/收伏笔可追踪 |
| **3** | Memory Layer | L0 Redis + L1 PG 摘要 + L2 pgvector；Memory Agent 检索路由 | 第 100 章能召回第 1 章相关片段 |
| **4** | Agent Runtime | 函数编排跑通 Story Engine 闭环 → 迁移 LangGraph StateGraph | 点「开始创作」产出带 reason/used_memory 的章节 |
| **5** | Skill Runtime | `skill.yaml` 加载器 + 现有 skillLibraryV2 迁移 + 组合链 | 加一个 `platform_xxx/` 目录即生效 |
| **6** | Token Governor | 预算层级 + 预估 + 压缩降级 + 成本核算 | 超预算自动压缩，成本可查 |
| **7** | 真实 LLM | ModelRouter 接真实 Provider（经 Governor 包装） | 脱离 Mock，产出真实文本 |
| **8** | 异步/工程化 | Celery/Dramatiq 任务队列、Event Bus、Swagger、单测、部署 | 长生成异步化，前端 SSE 实时进度 |

> **关键决策（沿用 v1 建议并强化）**：
> 1. **Milvus 太重** → 开发期 `pgvector`，生产 Phase 8 切 Milvus。
> 2. **LangGraph 后上** → Phase 4 先函数编排，降风险。
> 3. **前后端解耦过渡** → 中间用 Next.js API Route 做适配层转发 FastAPI，Phase 8 切净。
> 4. **事件溯源是地基** → Phase 1 必须扎实，后续全依赖它。

## 11. 一句话收尾

v1.0 把项目从「小说 CMS」拉到「AI 创作平台」；v2.0 把它从「AI Writer」升到 **Story Engine**——

这，才是「百万字不崩」的真正答案。
