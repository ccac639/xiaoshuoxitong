# 后端迁移执行计划 · AI Novel OS Lite（v2.1 → v2.4）

> 依据：`docs/架构设计-v2.0-Lite-个人版.md`
> 范围：将 `xiaoshuoxitong` 后端从「页面驱动 CRUD」迁移为「AI Novel OS Lite 内核」，**不重写前端**。
> 状态：本文件为执行版计划，供代码助手分阶段落地。

---

## 0. 一句话目标与硬约束

**目标**：让系统支撑 100 万字长篇写作，第 1000 章仍能稳定回忆第 1 章剧情（事件溯源 + 分层记忆 + 向量检索）。

**硬约束（不可违反）**：
1. ❌ 不重写前端页面/组件；保留现有资产，尤其是三大核心 Panel。
2. ❌ 前端不得直接 `fetch` 任何 LLM 端点，不得硬编码写作 Prompt。
3. ❌ 不推倒现有后端逻辑（现有 `src/server/**` 是资产，要**下沉/迁移**而非重写）。
4. ❌ 不在 Phase 2 之前引入 LangGraph / Milvus（按计划先用纯 Python 函数 + pgvector）。
5. ✅ 所有 AI 调用统一走：`前端 → 后端 API → Agent → Skill → LLM`。

## 1. 现状基线（已核实仓库）

**前端（保留）**
- Next.js 14（含 `app/` + `pages/`）。
- 已确认资产：`src/components/world-state/WorldStatePanel.tsx`、`src/components/memory/StoryMemoryView.tsx`。
- 待核对真实文件名：`Event Flow Panel`、`Skill Runtime Panel`（实施 v2.1 时 `grep` 组件名确认，勿凭假设）。

**现有后端逻辑（待下沉，非重写）**
| 现有文件 | 在 Lite 内核中的归宿 |
|---|---|
| `src/server/generation/modelRouter.ts` | → Token Governor 的执行臂（模型路由 + 成本核算） |
| `src/server/ai/aiClient.ts`（多 Provider：SiliconFlow / 智谱 / OpenRouter / 移动云 MoMA） | → 后端 `agent/` 的统一 LLM 客户端 |
| `src/server/generation/xiaoshuoProxy.ts` | → 后端 `api/` 的统一入口范式 |
| `src/server/world/stateManagerV2.ts` | → World State 投影缓存 |
| `src/server/skills/skillLibraryV2.ts` | → Skill Runtime（→ `skill.yaml` + `prompt.md`） |

**现状问题**：页面驱动 CRUD，无事件溯源、无分层记忆、无 Token 治理、无向量检索。

## 2. 目标架构

新增**独立 `backend/` 目录**（与现有 Next 前端解耦，通过 REST API 通信）：

```
backend/
├── api/          # FastAPI 路由层（对齐现有 xiaoshuoProxy 的入口范式）
├── core/         # 配置、DB 连接、依赖注入、统一 LLM 客户端（复用 modelRouter 逻辑）
├── models/       # SQLAlchemy ORM 模型（6 张表）
├── agent/        # Agent Runtime：Planner / Writer / Audit 三阶段编排
├── memory/       # 分层记忆 L1/L2/L3 + pgvector 向量检索
├── event/        # Story Engine：事件溯源 + World State 投影
├── skill/        # Skill Runtime：加载 skill.yaml + prompt.md 并执行
└── token/        # Token Governor：路由 + 成本核算 + 用量落库
```

**基础设施**：PostgreSQL（+ pgvector 扩展）、Redis（缓存 / 任务队列）、Docker（docker-compose 一键起）。

---

## 3. 数据库第一版（仅 6 张表）

人物 / 关系暂用 `world_state` 的 `JSONB` 存储，不单独建表。

| 表 | 关键字段 | 职责 |
|---|---|---|
| `novel` | `id, title, premise(JSONB), meta(JSONB), created_at, updated_at` | 作品本体 |
| `chapter` | `id, novel_id(FK), chapter_no, title, content(text), outline(JSONB), status, created_at` | 章节正文与大纲 |
| `story_event` | `id, novel_id(FK), chapter_id(FK NULL), event_type, payload(JSONB), embedding(vector, pgvector), created_at` | **事件溯源（唯一真相）** |
| `world_state` | `id, novel_id(FK), key, value(JSONB), version, updated_at` | **世界状态投影**（人物/关系存于 `key='characters'/'relations'`） |
| `skill_execution` | `id, novel_id(FK), skill_name, input(JSONB), output(JSONB), status, created_at` | Skill 执行记录 |
| `token_usage` | `id, novel_id(FK NULL), provider, model, prompt_tokens, completion_tokens, cost, created_at` | Token 治理 / 成本统计 |

> 核心范式：**事件溯源是唯一真相，`world_state` 是投影，记忆是检索层**。状态不靠“覆盖更新”，靠“重放事件”。

## 4. 分阶段迁移路线（按优先级执行）

### v2.1 — 后端骨架
- **目标**：跑通 FastAPI + Postgres + Redis + Docker，建立 6 表与基础 CRUD。
- **产出**：
  - `backend/` 完整目录骨架 + `docker-compose.yml`（Postgres+Redis+pgvector）。
  - 6 张表的 SQLAlchemy 模型 + Alembic 迁移脚本。
  - `api/` 基础路由：`novel` / `chapter` 的 CRUD + `/healthz`。
  - `core/` 配置与 DB 连接；`core/llm_client.py` 复用 `modelRouter` 的多 Provider 路由逻辑。
- **验收**：`docker compose up` 后，`POST /api/novel` + `POST /api/chapter` 可建读；**现有前端仍能正常运行**（未改前端）。
- **风险**：Docker/网络环境差异；Postgres pgvector 扩展需先 `CREATE EXTENSION`。

### v2.2 — Story Engine（Event + World + Memory，pgvector）
- **目标**：建立事件溯源 + 世界状态投影 + 分层记忆与向量召回。
- **产出**：
  - `event/`：写 `story_event`（事件溯源）、由事件重放生成 `world_state` 投影（下沉 `stateManagerV2`）。
  - `memory/`：L1 近期上下文 / L2 章节摘要 / L3 长期向量记忆；pgvector `embedding` 列检索（下沉 `StoryMemoryV2` 思路）。
  - `api/` 新增：`/api/event`、`/api/world-state`、`/api/memory/recall`。
- **验收**：写入第 1 章事件后，在第 N 章调用 `/api/memory/recall` 能向量召回第 1 章关键剧情。
- **风险**：向量维度需与嵌入模型一致；大量事件重放的性能（先用索引，后期再优化）。

### v2.3 — AI Loop（Planner → Writer → Audit）
- **目标**：实现「大纲规划 → 章节写作 → 质量审计」三阶段流水线。
- **产出**：
  - `agent/` 先用**纯 Python 函数**实现三阶段（复用现有 `ChapterWriterV2` 六阶段思路），经 `core/llm_client` 调用统一 LLM。
  - `api/` 新增：`/api/write`（给定 novel+大纲产出一章）、`/api/audit`（评分反馈）。
  - 后续（v2.3.x）再将纯函数编排替换为 LangGraph（不提前引入）。
- **验收**：给定大纲自动产出一章，audit 给出稳定性评分；质量可复现、不崩。
- **风险**：长文一致性；Audit 标准需可配置；超时/降级（参考现有 aiClient 的异常兜底）。

### v2.4 — Skill 系统
- **目标**：把写作辅助能力外置为 `skill.yaml` + `prompt.md`，由 Skill Runtime 加载执行。
- **产出**：
  - `skill/`：Skill Runtime 加载机制；首批 skill：`fanqie`（番茄风）、`ai_remove`（去 AI 味）、`audit`（审计）。
  - 每个 skill = `skill.yaml`（元信息/参数）+ `prompt.md`（提示词模板），下沉 `skillLibraryV2`。
  - `api/` 新增：`/api/skill/{name}` 触发执行，结果入 `skill_execution` 表。
- **验收**：通过 API 触发任一 skill，返回结构化结果且落库。
- **风险**：prompt 模板版本管理；skill 间依赖（先单 skill，后组合）。

## 5. 前端改造映射

将前端改造为 **Dashboard + Chapter Writer + Event Flow Panel** 三核心页：

| 改造项 | 动作 |
|---|---|
| 直接调用 LLM 的代码 | 删除，改为请求后端 `/api/write`、`/api/audit` 等 |
| 硬编码写作 Prompt | 删除，Prompt 收归后端 `skill/*.md` 与 `agent/` |
| Event Flow Panel | 保留并改连 `/api/event`、`/api/world-state`（数据源自后端） |
| World State Panel | 保留（`WorldStatePanel.tsx` 已确认），改连 `/api/world-state` |
| Skill Runtime Panel | 保留并改连 `/api/skill/*` |
| 新增 Dashboard | 统揽 novel / 进度 / Token 用量（`token_usage` 表） |

**统一链路**：`前端 → 后端 API → Agent → Skill → LLM`（前端永不见 API Key 与 Prompt）。

---

## 6. 执行原则（防回退）

1. **迁移非重写**：现有 `src/server/**` 资产下沉到 `backend/`，不推倒重来。
2. **每阶段独立可验、可回滚**：v2.1 完成前不碰 v2.2；任一阶段失败可退回上一稳定态。
3. **数据库变更走 Alembic 迁移脚本**，禁止手工改表。
4. **前后端解耦**：先做适配层，前端逐页切到新 API，不一次性大改。
5. **敏感信息**：LLM API Key 仅存后端环境变量，前端零密钥。

- [ ] `token_usage` 表可统计各 Provider / 模型成本。
- [ ] 三大核心 Panel 数据全部源自后端 API。
